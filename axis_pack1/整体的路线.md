# 你当前管线的“最可能真实形态”（全链路）

## A. 输入与分发层（入口）

### 1) 输入读取

- 读取脚本/台词/标注文件（可能是 CSV/JSONL/TXT）
- `line_selector` 在这里生效：只挑你指定的行/段落跑

### 2) axis 维度拆分

- 你会对同一份文本，按 axis 生成不同输出（比如：不同角色/不同风格/不同后处理策略/不同采样率规范等）
- 每个 axis 都写自己的 manifest：`manifest_fps[axis] = ...`

> **manifest 的角色**：记录每条输入最终产出音频、元数据（emotion、intent、micro 参数、时长、失败原因、重试次数等）。
> 如果你要“全面看你现在到底做了什么”，manifest 就是第一现场。

------

## B. 文本处理层（Text preprocessing）

这层通常在 `conv` 或某个 `text_processor` 里做，常见包含：

1. **基础清洗**
   - 去多余空白、统一全半角、特殊符号替换
2. **规范化（Normalization）**
   - 数字、日期、金额、单位（“12kg/3.5万/2024-01-01”读法）
   - 英文/缩写读法（AI、CEO、URL）
3. **分句/断句**
   - 按标点、长度阈值、括号/引号结构切段
4. **注入控制符**
   - 插入停顿（pause）、强调（emphasis）、语速/情绪提示（如果你用 prompt 或标签）
5. **对话结构抽取（如果输入是 conversation 脚本）**
   - 谁说话（speaker）
   - 轮次顺序
   - 旁白/舞台提示（可选是否朗读）

**你可以在代码里用关键词定位这层：**

- `normalize`, `clean`, `text_process`, `split`, `segment`, `punct`, `ssml`, `pause`, `tokenize`

------

## C. 意图层（Intent）

你说的“意图（包含什么）”很关键：意图通常不是模型的声学参数，而是**业务语义到“生成策略”的路由器**。

常见 intent 会决定：

- 用哪个 speaker / style（比如客服/播报/闲聊）
- 是否更正式、更快、更“明确咬字”
- 是否走某个特殊模板（播报类：强断句、低情绪；陪伴类：更软、更慢）
- 是否启用某些“异常治理”（比如语速异常/时长不符合预期就重试）

**在你这种工程里，intent 很可能长这样：**

- `intent = intent_classifier(text, meta)` 或 `intent = rule_based(text)`
- 然后映射到一个策略对象：`strategy = INTENT2STRATEGY[intent]`

**定位关键词：**

- `intent`, `route`, `strategy`, `policy`, `scenario`, `domain`

------

## D. Emotion 标签路径（你问的“emotion 走什么路径”）

在“封装推理”架构里，emotion 通常有三种落点（你可能是其中一种或混合）：

### 路径 D1：emotion → **prompt/条件输入** → IndexTTS2

- 你把 emotion（如 happy/sad/angry/neutral）映射成：
  - 一段风格提示词 / style token
  - 或一个可控参数（`emotion_id`、`style_id`、`ref_audio` 选择）
- 然后在调用 `IndexTTS2` 的 `infer/synthesize` 时传进去

**特征：**

- emotion 改变会显著影响韵律/音高/能量分布

**定位关键词：**

- `emotion`, `style`, `prompt`, `condition`, `ref`, `embedding`, `emotion_id`

### 路径 D2：emotion → **后处理参数集**（更像“调音台”）

- emotion 不直接进模型，而是决定后处理：
  - 伤心：整体能量低一点、尾音更长、停顿更长
  - 生气：能量更高、语速更快、动态范围更大
- 这条路更“工程化”，但也更容易把自然度搞坏（后处理太狠会假）

**定位关键词：**

- `loudness`, `gain`, `compress`, `pause_ms`, `silence`, `fade`, `prosody_post`

### 路径 D3：emotion → **文本侧控制符**

- 通过插标点、插 pause、插强调标签实现（常见于不支持显式 emotion 控制的模型）

**定位关键词：**

- `pause`, `break`, `ssml`, `prosody`, `emphasis`

> 你问“emotion 标签走的是什么路径”，本质就是：**D1（进模型）/D2（后处理）/D3（文本注入）到底是哪一种**。
> 你项目里很可能至少有 D1 或 D3（因为你已经有比较完整的 config/conv 层）。

------

## E. Micro 路径（“micro 走什么路径？”）

“micro”在 TTS 工程里最常见的含义是：**微韵律控制**（micro-prosody），比如更细粒度的停顿、重音、局部语速变化、句尾上扬/下落、关键词强调等。

它同样有三种常见落点：

### 路径 E1：micro → **文本标注/重写**

- 微停顿：在特定词后加逗号/顿号/停顿标记
- 关键词强调：插 emphasis
- 句尾语气：改问号/感叹号/省略号策略

### 路径 E2：micro → **推理参数**（duration/pitch/energy 类）

- 如果你的 IndexTTS2 或其 wrapper 暴露了控制项（不确定你有没有接）
- micro 可能会变成 `duration_scale`, `pitch_shift`, `energy_scale` 之类

### 路径 E3：micro → **音频后处理**

- 这最危险：后处理很难“伪造”自然微韵律，通常只会变得塑料
- 但也可能用于做很轻的尾音延长、停顿插入、淡入淡出，算“微观层面的编辑”

**定位关键词：**

- `micro`, `prosody`, `duration`, `pitch`, `energy`, `emphasis`, `break`, `pause_map`

------

## F. 语速异常路径（你点名的“语速异常这一部分”）

语速异常通常不是一个“固定参数”，而是一个**治理闭环**。典型闭环长这样：

1. **预测期望时长**（基于字数/音素数/语言类型/标点）
2. **生成音频并测量真实时长**
3. **比对阈值**
   - 太快：真实时长 << 期望时长
   - 太慢：真实时长 >> 期望时长
4. **采取动作**
   - 调 `speed`/`rate` 参数重试
   - 或改文本（插 pause/增强断句）重试
   - 或换策略（换 voice/style）
   - 或直接标记异常写入 manifest

在你的工程里，这个模块可能在：

- `post_check` / `quality_check` / `abnormal_detector`
- 或者 manifest 生成时就会写入字段：`speed_abnormal=true`

**定位关键词：**

- `speed`, `rate`, `duration`, `sec`, `len`, `abnormal`, `retry`, `threshold`, `qc`, `quality`

# conversation 总音频是怎么生成的（路径复盘）

你提到“生成整体音频 conversation 的时候我怎么走路径”。

从你那句 `args.conversation_only -> args.with_conversation = True` 来看，你的 conversation 生成至少有一个开关：`with_conversation`。

最常见（也最贴合你 axis/manifest 的）实现路径是：

1. **对话脚本解析**
   - 抽取 turns：`(speaker, text, emotion?, micro?, intent?)`
2. **逐 turn 走“单句生成链路”**
   - 文本处理 → 意图路由 → emotion/micro 参数注入 → IndexTTS2 生成 → 单句后处理 → 保存 turn wav
3. **conversation 合成**
   - 读取这一段对话所有 turn wav（按顺序）
   - 在 turn 之间插入：
     - 固定静音（比如 150ms）
     - 或按标点/情绪决定的可变静音
     - 或 crossfade（避免拼接“咔哒”）
   - 可选：整体 loudness/峰值限制（limiter）
4. **写出 conversation.wav + conversation manifest**
   - 通常会额外写：总时长、每 turn 起止时间（便于字幕/对齐）

**定位关键词：**

- `conversation`, `with_conversation`, `conversation_only`
- `concat`, `merge`, `join`, `append`
- `silence`, `crossfade`, `fade`, `interval`
- `turn`, `utterance`, `dialog`, `speaker`

# 你要“全面取舍”，我给你一张工程取舍地图（按模块）

## 1) emotion：进模型 vs 文本注入 vs 后处理

- **进模型（D1）**：效果最好、风险中等（取决于模型支持度）
- **文本注入（D3）**：最稳、最可控，但上限有限
- **后处理（D2）**：最容易出塑料味，慎用，只做轻量

## 2) micro：文本标注最值钱

- micro 做成“文本侧规则”通常收益最高（可解释、可回归）
- micro 走音频后处理收益最低且风险最高

## 3) 语速异常：不要只靠“调速参数”

- 纯调 `speed` 很容易造成发音质量变差或韵律崩
- 更可靠的做法是：**先改文本断句/停顿**，再必要时微调 speed，然后再重试

## 4) conversation 合成：问题大多来自“拼接学”

- 拼接点 click/pop
- turn 间静音太死板导致节奏怪
- 多 turn 合成后峰值爆炸（需要整体 limiter/归一化）

## 1) 根因：变量根本没初始化就被使用（直接崩）

### A. `emo_alpha` 在赋值前就被用到了

你在“场景行为调节”处写了：

- `apply_conversation_progression(... base_emo_alpha=emo_alpha ...)`

### B. `emo` 根本没定义

你前面把 `masked_labels()` 的返回接成了：


后面却大量使用 `emo`（比如安全阀、模板、build_emo_text/vector）。于是第二个报错 `name 'emo' is not defined` 立刻出现。

------

## 2) 第二层硬 bug：文件变量名错乱（就算推理成功也会写不出/写错）

你在同一个函数里混用了这些名字：

- 前面定义：`raw`、`out`
- 后面写：`raw_wav`、`out_wav`
- 语速阶段写：`speed_adjusted`（没定义）

典型的会导致：

- `audio_all.export(raw_wav, ...)` 但 `raw_wav` 未定义
- `os.path.getsize(speed_adjusted)` 但 `speed_adjusted` 未定义

这些都会让流程在“保存/后处理”阶段炸掉，最终依然没音频。
